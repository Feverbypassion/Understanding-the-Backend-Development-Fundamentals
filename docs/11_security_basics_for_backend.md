# 11. Security Basics for Backend

## 11.1 How to Read This Chapter

Infrastructure connects a backend to the outside world. Security helps us decide what a backend can trust, what it must verify, and what it needs to protect once requests start arriving.

This chapter is a concept-first introduction to backend security. You are not expected to implement a production authentication system, OAuth flow, payment verification system, or anti-cheat system here. Focus on learning the questions that help a backend handle player data, rewards, scores, rankings, and admin operations safely.

The most important idea in this chapter is simple.

```text
A backend should not blindly trust the client, especially when player data, rewards, scores, or rankings are involved.
```

This does not mean that every player is malicious. It means that the client runs in an environment controlled by the player. Game clients, mobile apps, PC builds, WebGL builds, browser JavaScript, local storage, device time, and network requests can be inspected, modified, replayed, or generated by custom tools.

At this stage, treat security as a way of thinking, not as a list of attack names to memorize. When you see an API, ask:

- Who sent this request?
- Is this user allowed to perform this action?
- What values came from the client?
- Which values should the server verify or decide?
- What happens if the same request is repeated many times?
- Could the response, error message, or log expose sensitive information?

We will use game backend examples such as score submission, daily reward claims, inventory lookup, leaderboard access, and admin actions. These examples are engine-independent and are meant to help you understand security decisions before you study implementation details in later courses.

## 11.2 What You Will Learn

By the end of this chapter, you should be able to explain the following ideas in your own words:

- Why backend security is connected to fairness, trust, and operational stability.
- Why client-sent values such as scores, reward IDs, currency amounts, and player IDs should not be trusted blindly.
- The difference between authentication and authorization.
- Why input validation includes format, range, and game-rule validation.
- Why repeated requests, bots, and duplicate reward claims can damage a game service.
- What HTTPS protects and what it does not replace.
- What CORS does for browser-based clients and why it is not a general API authentication or authorization system.
- Why secrets should not be placed in source code, Git repositories, game clients, or browser JavaScript.
- How OWASP API Security Top 10 2023 can be used as a beginner-friendly risk map.
- Why server authority matters for scores, rewards, currency, rankings, and match results.

You should also understand what this chapter does not cover in depth. Production authentication, OAuth, payment validation, anti-cheat systems, admin role management, and full incident response belong to later advanced courses.

## 11.3 Why Security Matters in Game Backends

Game backend security is not only about blocking attackers from the outside. A game backend is tied to player accounts, time, effort, purchases, rewards, rankings, event participation, and operator actions. If these are processed incorrectly, the problem can affect both player trust and live operations.

Imagine that a player clears a stage and submits a score.

```json
{
  "stageId": "stage_01",
  "score": 999999999,
  "playTimeSeconds": 3
}
```

The request may be valid JSON, but that does not mean the result is believable. The server needs to ask whether the score is possible for the stage, whether the player was allowed to enter that stage, whether the request is being repeated too often, and whether the score should be accepted into the leaderboard.

Now imagine a daily reward request.

```http
POST /rewards/daily
Authorization: Bearer player-token
```

The client may only be asking to claim today's reward, but the backend must decide whether the player has already claimed it, which server-defined day boundary or reset period should be used, which reward rule applies, and whether the inventory update has already happened. Without those checks, repeated requests can grant duplicate rewards.

Security protects at least three things in a game backend.

| Perspective | What It Protects | Game Backend Example |
|---|---|---|
| Fairness | Players compete under the same rules. | Scores, rankings, match results, and season rewards are checked. |
| Trust | Player progress and value are handled safely. | Accounts, inventory, currency, purchases, and reward history are protected. |
| Operational Stability | The team can investigate and recover from problems. | Admin actions, audit logs, safe errors, and clear records support operations. |

A common beginner mistake is thinking, “We can add security later.” In practice, security starts when you design the first API shape. If an API allows the client to decide too much, it may be difficult to make it safe later.

## 11.4 What a Backend Must Protect

The starting point of security is knowing what must be protected. Not all backend data carries the same level of risk. A public leaderboard API is different from an admin API that grants items to players.

The table below shows common protected targets in game backends.

| Protected Target | Why It Matters |
|---|---|
| Account | It identifies the player and connects saved data. |
| Session or Token | If exposed, someone else may impersonate the player or operator. |
| Inventory | It records items, equipment, cards, and reward history. |
| Currency | It affects progression, economy balance, shop behavior, and sometimes purchases. |
| Score and Ranking | They affect competition, fairness, and season rewards. |
| Reward State | It prevents duplicate grants and supports customer support checks. |
| Event Config | It can change LiveOps results, schedules, and rewards. |
| Admin API | It can modify live data and operational settings. |
| Secrets | Database passwords, API keys, and token signing secrets must not be exposed. |

These protected targets are accessed through APIs. That means high-risk data usually creates high-risk API design questions.

Consider these APIs:

```http
GET  /me/inventory
POST /scores
POST /rewards/daily
GET  /leaderboard
POST /admin/events
GET  /admin/audit-logs
POST /admin/players/{playerId}/grant-item
```

Each API raises a different kind of question.

| API | First Security Question |
|---|---|
| `GET /me/inventory` | Does it return only the authenticated user's own inventory? |
| `POST /scores` | Is the score possible, and is the submission frequency suspicious? |
| `POST /rewards/daily` | Has the player already claimed today's reward according to server state? |
| `GET /leaderboard` | Can this be public, or does it need caching and rate limiting? |
| `POST /admin/events` | Is the requester an authorized operator, and will the change be audited? |
| `GET /admin/audit-logs` | Who is allowed to view sensitive operational records? |
| `POST /admin/players/{playerId}/grant-item` | Is this action permitted, confirmed, validated, and audit-logged? |

For beginners, this is already a strong starting point: before asking whether an API “works,” ask what could go wrong if the API is called by the wrong user, with wrong values, too often, or at the wrong time.

## 11.5 Core Concepts

This section introduces the core concepts that appear repeatedly in backend security. You do not need to implement them in this chapter. The goal is to understand what problem each concept helps solve.

In this section, we will look at seven core ideas:

- Client-sent values and trust
- Authentication and authorization
- Input validation and abuse prevention
- HTTPS, CORS, and secret management
- OWASP API Security Top 10 2023 as a risk map
- Server authority
- Safe error responses and sensitive logs

### 11.5.1 Do Not Trust Client-Sent Values Blindly

The client runs close to the player, on a device the player controls. The server is the reference point for important service data.

That means the client can present UI, collect input, and show effects, but the backend should verify or decide values that affect fairness, economy, rankings, rewards, or player data.

A dangerous score submission might look like this:

```json
{
  "playerId": "player_001",
  "stageId": "stage_01",
  "score": 999999999,
  "rewardGems": 5000
}
```

There are two problems here. First, the score may be impossible. Second, the client is trying to send the reward amount directly. A safer design asks the client to submit the result or claim, while the server checks the rule and decides what reward applies.

```http
POST /scores
```

```json
{
  "stageId": "stage_01",
  "score": 15200,
  "playTimeSeconds": 94
}
```

In this safer shape, the backend can check the authenticated player, stage access, possible score range, duplicate submission, and ranking policy. The server should not treat a client field as authoritative simply because it appears in JSON.

This shape is safer than letting the client decide rewards or currency changes, but it is still not a complete anti-cheat design. In highly competitive games, the server may need stronger evidence, replay checks, server-side simulation, or authoritative match logic, often running on a Dedicated Game Server.

A useful question is:

```text
If this value were manipulated, would it affect fairness, economy, player trust, or operations?
```

If the answer is yes, the backend should verify it carefully or decide it on the server side.

### 11.5.2 Authentication and Authorization

Authentication and authorization are related, but they are not the same.

```text
Authentication: Who sent this request?
Authorization: Is this user allowed to do this?
```

A user may be authenticated through guest login, platform login, email login, social login, or another account flow. After authentication, the backend usually has some way to identify the requester, such as a session, token, or server-side account record.

Authorization asks a different question. A user may be logged in, but that does not mean they can access every record or call every API.

| Situation | Authentication Question | Authorization Question |
|---|---|---|
| View inventory | Who is the requester? | Is this the requester's own inventory? |
| Submit score | Which player sent the request? | Can this player submit a score for this stage and account? |
| Claim daily reward | Which player is claiming? | Is this player allowed to claim today's reward now? |
| Change event settings | Which operator is logged in? | Does this operator have event management permission? |
| Grant an item | Which operator is acting? | Does this role allow item grants under this condition? |

A common risky API shape is this:

```http
GET /players/player_002/inventory
Authorization: Bearer token-of-player-001
```

If the token belongs to `player_001`, the server must not return `player_002`'s inventory just because the path says so. A safer design for a player viewing their own data is:

```http
GET /me/inventory
Authorization: Bearer token-of-player-001
```

Here, the server reads the user identity from the verified token or session and returns only that user's inventory.

At this stage, you are not expected to implement JWT validation, session storage, OAuth login, or password recovery. Remember the concept: every sensitive API needs both identity verification and permission checking.

### 11.5.3 Input Validation and Abuse Prevention

Input validation checks whether values sent by the client match what the server expects and allows. Beginners often think validation only means checking types, but game backend validation goes further.

Input validation usually includes three layers.

| Layer | Question | Example |
|---|---|---|
| Format validation | Is the type or format correct? | Is `score` a number? Is `rewardId` a string? |
| Range validation | Is the value within an allowed range? | Is the score too large? Is the nickname too long? |
| Rule validation | Is this action possible under game rules? | Can this player claim this reward now? |

A request can pass the first layer but still be dangerous.

```json
{
  "currencyId": "gem",
  "amount": 100000
}
```

The value `100000` is a number, so it may pass format validation. But if a regular player can decide how many gems should be granted, the API design is unsafe. A better shape is to let the server decide the reward amount from server-side rules.

```http
POST /rewards/daily
```

The client asks to claim the daily reward. The server checks server time, player state, reward configuration, and claim history. Then the server decides whether to grant the reward and how much to grant.

Abuse prevention is also part of safe backend thinking. Even a normal API can become harmful if it is called too frequently.

| API | Abuse Risk | Possible Protection Direction |
|---|---|---|
| Login | Password guessing or automated attempts | Limit repeated attempts and monitor failures. |
| Score submission | Repeated suspicious scores | Limit frequency and check game-rule validity. |
| Daily reward claim | Duplicate grant attempts | Check server-side claim state and consider idempotency. |
| Leaderboard read | Excessive traffic | Cache responses and apply reasonable limits. |
| Admin API | Repeated risky changes | Require permission checks, confirmation, and audit logs. |

Rate limiting can reduce repeated requests, but it does not make an API safe by itself. Rate limits can be applied using different keys, such as IP address, authenticated user, device, token, endpoint, or a combination of these. The right choice depends on what kind of abuse the team is trying to reduce.

A reward API still needs server-side claim state. A score API still needs score validation. An admin API still needs authorization.

For duplicate-sensitive features such as rewards, payments, currency grants, and inventory updates, you may also hear the word idempotency. Idempotency means the server is designed so that the same request does not apply the same change multiple times. In real systems, this may use a request ID, transaction ID, or idempotency key, but this chapter only introduces the idea.

### 11.5.4 HTTPS, CORS, and Secret Management

HTTPS, CORS, and secret management often appear together, but they protect different things.

| Topic | Basic Role | What It Does Not Replace |
|---|---|---|
| HTTPS | Protects data in transit between client and server. | It does not replace authentication, authorization, or validation. |
| CORS | Tells the browser which cross-origin responses a browser-based frontend is allowed to read. | It does not replace authentication, authorization, validation, or CSRF protection. |
| Secret management | Protects server-only values such as database passwords and signing keys. | It does not make client-embedded values secret. |

HTTPS protects communication while data moves over the network. Login tokens, session cookies, request bodies, and response data should not travel through plain HTTP in a real service.

CORS, or Cross-Origin Resource Sharing, is enforced by browsers. It tells the browser which cross-origin responses a browser-based frontend may read. This matters for browser-based clients and admin frontends.

For example, if an admin frontend runs at one origin and the API runs at another origin, the browser checks whether that frontend is allowed to read the API response.

However, CORS is not a general API authentication or authorization system. Non-browser tools such as curl, Postman, native game clients, and custom scripts generally do not rely on browser CORS enforcement, so the backend must still verify identity, permissions, and input values on the server side.

If a browser-based admin frontend uses cookie-based authentication, CSRF protection is a separate topic to consider. This matters because browsers may automatically include cookies with requests, depending on cookie settings and the request context. CORS settings do not automatically solve CSRF risks.

Secret management is about protecting values that must not be exposed.

Examples of secrets include:

- Database passwords
- Token signing secrets
- Internal admin keys
- Server-to-server API keys
- External service secret keys

A bad example is hardcoding secrets in source code.

```js
const DATABASE_URL = "postgres://admin:password123@example.com/game";
const INTERNAL_ADMIN_KEY = "my-secret-admin-key";
```

A better direction is to keep real values outside the source code and provide only placeholders in `.env.example`.

```env
DATABASE_URL=replace-with-your-database-url
TOKEN_SECRET=replace-with-your-token-secret
INTERNAL_ADMIN_KEY=replace-with-your-internal-admin-key
```

The real `.env` file should not be committed to Git. If a secret has already been committed to a repository, deleting the line is usually not enough. Treat the secret as exposed and revoke, rotate, or replace it.

Also remember this rule:

```text
Anything embedded in a game client, WebGL build, or browser JavaScript should be treated as public.
```

Hard to find is not the same as secret. If the server alone should know a value, do not put that value in the client.

### 11.5.5 API Security and OWASP as a Risk Map

The OWASP API Security Top 10 is updated over time. In this chapter, we use the API Security Top 10 2023 edition as a beginner-friendly map of API-specific risks. This is separate from the OWASP Top 10 for general web application risks. At this stage, you do not need to memorize every item. Use it to organize the questions you should ask when reviewing APIs.

Here are a few OWASP-style risks connected to game backend examples.

| Risk Area | Game Backend Example |
|---|---|
| Broken Object Level Authorization | A player can view another player's inventory by changing a player ID in the path. |
| Broken Function Level Authorization | A regular user calls an admin API such as `POST /admin/events`. |
| Broken Object Property Level Authorization | The server accepts client-supplied fields such as `isAdmin`, `role`, or `gemBalance`. |
| Unrestricted Resource Consumption | Login, search, leaderboard, or score APIs can be called too frequently. |
| Unrestricted Access to Sensitive Business Flows | Automated reward claiming, coupon use, or account creation is abused. |
| Security Misconfiguration | Debug information, development settings, or internal errors are exposed. |
| Improper Inventory Management | Old API versions, undocumented endpoints, exposed test environments, or forgotten integrated services remain open or poorly protected. |

In OWASP terminology, “inventory” means the documented map of API hosts, environments, versions, endpoints, parameters, requests, responses, and connected services. It does not mean a player's item inventory in a game.

Use OWASP as a study map, not as a memorization list. When you see a game backend API, connect the abstract risk to a concrete game situation.

### 11.5.6 Server Authority in Game Backends

Server authority means that important game outcomes are verified or decided by the server instead of being freely decided by the client.

It does not mean the server must simulate every animation, button, camera effect, or UI transition. The client can handle presentation. The server should protect important outcomes.

| Feature | What the Client Can Do | What the Server Should Check |
|---|---|---|
| Reward animation | Show a reward box opening animation. | Whether the reward is actually granted. |
| Score screen | Show the score after a stage. | Whether the score is possible and accepted. |
| Daily reward button | Send a claim request. | Whether today's reward can still be claimed. |
| Shop purchase UI | Show item and price information. | Balance, price, purchase record, and grant result. |
| Match result screen | Show victory or defeat. | Final result, reward eligibility, and ranking impact. |

Server authority and anti-cheat are related, but they are not the same. Server authority is a design approach. Anti-cheat is broader and may include prevention, detection, monitoring, review, and penalties or account actions.

Different games require different levels of server authority. A casual single-player game and a real-time competitive game have different risk profiles. But data that affects rewards, currency, rankings, match results, and player trust should be verified or decided by the server whenever possible.

A beginner-friendly rule is:

```text
The client may present effects and UI.
The server should verify important rewards and outcomes.
```

### 11.5.7 Safe Error Responses and Sensitive Logs

Security is also connected to error handling and logs.

A client response should be helpful enough for the client to react, but it should not expose internal details such as stack traces, database queries, file paths, server environment values, or secrets.

A risky error response might look like this:

```json
{
  "error": "DatabaseError",
  "query": "SELECT * FROM users WHERE token = 'secret-token-value'",
  "file": "/app/src/admin/grantItem.js",
  "stack": "..."
}
```

A safer client-facing response is shorter and can include a request ID.

```json
{
  "error": "RewardClaimFailed",
  "message": "The reward could not be claimed. Please try again later.",
  "requestId": "req_9f21c0"
}
```

A request ID helps support staff and developers find the related server log without exposing stack traces, queries, tokens, or file paths to the client.

Server logs may contain more detail for debugging and operations, but they should still avoid recording sensitive values directly. Avoid logging passwords, access tokens, refresh tokens, API keys, raw payment receipts, or unnecessary personal data.

The next chapter builds on this idea: logs and metrics should help the team understand what happened, but they must be designed so that tokens, secrets, and unnecessary personal data are not exposed.

## 11.6 Example Scenario: Score Submission and Daily Reward Claim

Let’s connect the concepts through two common game backend scenarios.

### 11.6.1 Scenario A: Score Submission

The player finishes a stage and the client sends a score request.

```http
POST /scores
Authorization: Bearer player-token
Content-Type: application/json
```

```json
{
  "stageId": "stage_01",
  "score": 15200,
  "playTimeSeconds": 94
}
```

The backend should not only ask whether the JSON is valid. It should ask several security questions.

| Check | Question |
|---|---|
| Authentication | Which player sent this request? |
| Authorization | Is this player allowed to submit a score for this account and stage? |
| Format validation | Are `stageId`, `score`, and `playTimeSeconds` in the expected format? |
| Range validation | Is the score within a possible range? |
| Rule validation | Is the score possible for the stage and play time? |
| Abuse prevention | Is the player submitting too frequently? |
| Server authority | Should the score be accepted, rejected, or flagged for review? |
| Logging | Can suspicious submissions be investigated later without exposing secrets? |

A simple introductory rule is that the server should reject clearly impossible values and record enough information to investigate suspicious ones.

This example helps you practice server-side checks, but it is still not a full authoritative result system.

Highly competitive games may need stronger evidence, replay checks, server-side simulation, or authoritative match logic, often running on a Dedicated Game Server. A complete anti-cheat system belongs in a later advanced course.

### 11.6.2 Scenario B: Daily Reward Claim

The player presses a daily reward button.

```http
POST /rewards/daily
Authorization: Bearer player-token
```

The request body can be empty because the client should not decide the reward amount.

```json
{}
```

The server should decide the reward based on server-side rules.

| Check | Question |
|---|---|
| Authentication | Which player is claiming? |
| Authorization | Can this player claim this reward for their own account? |
| Server time | What does “today” mean according to the server? |
| Claim state | Has this player already claimed today's reward? |
| Reward rules | Which reward applies today? |
| Idempotency | If the same request is repeated, will the reward be granted only once? |
| Inventory update | Was the item or currency granted safely? |
| Audit or operation record | Can support staff check the result later? |

A risky design lets the client send the reward amount.

```json
{
  "rewardId": "daily_01",
  "currencyId": "gem",
  "amount": 5000
}
```

A safer design lets the client ask to claim, while the server decides whether to grant a reward and which reward to grant.

```text
Client: I want to claim today's reward.
Server: I will check the player, date, claim history, reward rule, and inventory update.
```

These two examples show the same security mindset: the client can request an action, but the server should verify identity, permissions, input values, server state, request frequency, and important outcomes.

## 11.7 Learning Practice

This Learning Practice is for observation and study. It is not a production-ready implementation.

### Goal

Classify common game backend APIs by security risk and identify what the server should verify before processing each request.

### Step 1: Review the API List

Read the API examples below.

| API | Short Description |
|---|---|
| `POST /scores` | Submit a stage score. |
| `POST /rewards/daily` | Claim today's login or attendance reward. |
| `GET /leaderboard` | Read leaderboard ranking data. |
| `GET /me/inventory` | Read the authenticated player's inventory. |
| `POST /admin/events` | Create or update event settings. |
| `GET /admin/audit-logs` | View records of operator actions. |
| `POST /admin/players/{playerId}/grant-item` | Grant an item to a player through an admin API. |

### Step 2: Classify the Risk

For each API, mark the risk as Low, Medium, High, or Very High. If the risk depends on the game design or data exposed by the API, you may also use a range such as Low to Medium or Medium to High. Use the following questions:

- Does this API affect player value, rewards, currency, or rankings?
- Could this API expose personal or sensitive data?
- Could repeated calls cause damage?
- Is the API admin-only?
- Would a wrong result require customer support or operational recovery?

### Step 3: Choose the Needed Checks

Choose which checks are likely needed for each API.

| Check Type | Meaning |
|---|---|
| Authentication | Verify who sent the request. |
| Authorization | Check whether the requester is allowed to do this. |
| Input validation | Check format, range, and game-rule validity. |
| Rate limiting | Reduce repeated or automated calls. |
| Idempotency | Prevent duplicate processing for repeated requests. |
| Audit log | Record who performed an important action and why. |
| Server authority | Let the server verify or decide important outcomes. |
| Logging or operation record | Keep enough safe records to investigate suspicious or important actions later. |

### Possible Review After You Try

After you classify the APIs yourself, compare your reasoning with the table below. Your answer does not need to match the table exactly, but the reasoning should be similar.

| API | Likely Risk | Important Checks |
|---|---|---|
| `POST /scores` | Medium to High | Authentication, authorization, validation, rate limiting, server authority, logging. |
| `POST /rewards/daily` | High | Authentication, authorization, validation, claim-state check, idempotency, server authority, reward history or operation record. |
| `GET /leaderboard` | Low to Medium | Caching, rate limiting, response size limits, suspicious score policy, minimal public profile data. |
| `GET /me/inventory` | Medium | Authentication, authorization, reasonable rate limits, minimal response data. |
| `POST /admin/events` | High | Admin authentication, least-privilege authorization, validation, confirmation, change reason, audit log. |
| `GET /admin/audit-logs` | High | Admin authentication, least-privilege authorization, filtering, redaction, sensitive data access logging. |
| `POST /admin/players/{playerId}/grant-item` | Very High | Admin authentication, least-privilege authorization, target player validation, item validation, confirmation, approval rule, idempotency, audit log. |

### What to Observe

Notice the following patterns:

- Read-only APIs can still have security concerns.
- Admin APIs are high-risk even if they are not player-facing.
- Reward and currency APIs need server-side state checks, not only rate limits.
- CORS does not replace authentication, authorization, validation, or CSRF protection for cookie-based browser flows.
- Server authority is especially important for scores, rewards, currency, rankings, and match results.
- Logs and audit logs help operations, but they must not expose secrets or tokens.

## 11.8 Common Mistakes

### Mistake 1: Thinking Authentication Is Enough

Authentication tells the server who the requester is. It does not automatically mean that the requester can access every record or perform every action. A logged-in player should not be able to view another player's inventory or call admin APIs.

### Mistake 2: Trusting Client-Sent Player IDs

If the client sends `playerId`, the server should not blindly use it as the authority. For many player-owned APIs, the server should get the player identity from the verified token or session and use that as the source of truth.

### Mistake 3: Validating Type but Not Meaning

A value can have the correct type and still be dangerous. `score: 999999999` may be a number, but it may be impossible. `amount: 100000` may be a number, but regular players should not decide currency grants.

### Mistake 4: Treating CORS as Complete API Security

CORS is important for browser-based clients, but it is not a general API authentication or authorization system. It controls whether a browser-based frontend can read cross-origin responses. Native clients, curl, Postman, and custom tools generally do not rely on browser CORS enforcement, so server-side checks are still required. For cookie-based browser admin flows, CSRF protection may be a separate concern.

### Mistake 5: Thinking HTTPS Solves Every Security Problem

HTTPS protects data in transit. It does not decide who the user is, whether the user has permission, whether a score is possible, or whether a reward was already claimed.

### Mistake 6: Putting Secrets in Code, Git, or Client Builds

Secrets should not be hardcoded or committed. They should also not be embedded in game clients or browser JavaScript if the server alone should know them. If a secret is exposed, revoke, rotate, or replace it.

### Mistake 7: Exposing Internal Details in Error Responses

Client responses should not reveal stack traces, database queries, file paths, tokens, or secret values. Keep client errors safe and minimal.

### Mistake 8: Forgetting That Security and Operations Are Connected

Security decisions affect operations. Admin actions need audit logs, suspicious score submissions may need investigation, and reward grants need records that customer support can check later.

The next chapter continues this idea through observability, logging, and operations. Logs should be useful for investigation, but they must not expose tokens, secrets, or unnecessary personal data.

## 11.9 Chapter Summary

In this chapter, we studied the basic security mindset needed for backend systems.

The core idea is:

```text
Do not blindly trust the client.
```

A backend should verify the requester, check permissions, validate input values, limit repeated requests when needed, protect secrets, avoid exposing internal information, and decide or verify important game outcomes.

You learned that authentication and authorization are different. Authentication asks who sent the request. Authorization asks whether that requester is allowed to perform the action. You also learned that input validation includes format, range, and game-rule checks.

HTTPS protects data in transit, but it does not replace server-side security checks. CORS controls whether a browser-based frontend can read cross-origin responses. It does not authenticate users, authorize actions, validate requests, or replace CSRF protection for cookie-based browser flows.

Secrets should be kept out of source code, Git repositories, game clients, and browser JavaScript.

For game backends, server authority is especially important. The client may show UI and effects, but the server should verify important outcomes such as scores, rewards, currency, rankings, and match results.

In the next chapter, we will study observability, logging, and operations. Security helps us decide what to protect. Observability helps us understand what is happening in a running backend.

Logs and metrics should help the team understand what happened. They must also be designed so that tokens, secrets, and unnecessary personal data are not exposed.

## 11.10 Quiz

### Question 1

Which statement best describes the difference between authentication and authorization?

A. Authentication asks who sent the request; authorization asks whether that user is allowed to perform the action.  
B. Authentication checks request speed; authorization encrypts network traffic.  
C. Authentication validates JSON format; authorization stores leaderboard data.  
D. Authentication is only for admin users; authorization is only for players.

**Answer: A**

**Explanation:**  
Authentication identifies the requester. Authorization checks whether that requester has permission to perform a specific action.

### Question 2

Why should a backend be careful with client-sent score values?

A. Scores are always private data and must never be stored.  
B. The client may send incorrect or manipulated values.  
C. HTTPS cannot be used with score APIs.  
D. Leaderboards never need server-side data.

**Answer: B**

**Explanation:**  
The client runs in the player’s environment, so score values should be treated as input that the server verifies before accepting.

### Question 3

Which example best shows input validation beyond simple type checking?

A. Checking that `score` is a number only.  
B. Checking that `score` is a number, is within a possible range, and matches the stage rules.  
C. Accepting every score because it came from the game client.  
D. Moving validation from the server to the UI only.

**Answer: B**

**Explanation:**  
Good validation includes type, range, state, ownership, and game-rule checks when they matter for the API.

### Question 4

Which statement best explains why idempotency can matter for a reward API?

A. It makes all rewards public.  
B. It prevents repeated requests from accidentally granting the same reward multiple times.  
C. It replaces the need for authentication.  
D. It makes CORS unnecessary.

**Answer: B**

**Explanation:**  
If a reward request is retried, the backend should avoid applying the same reward effect multiple times.

### Question 5

Which statement about HTTPS and CORS is most accurate?

A. HTTPS protects communication in transit, while CORS controls browser cross-origin response access.  
B. HTTPS and CORS both replace authorization.  
C. CORS encrypts all game client traffic.  
D. HTTPS makes every client-sent value trustworthy.

**Answer: A**

**Explanation:**  
HTTPS and CORS solve different problems. Neither replaces server-side authentication, authorization, or validation.

### Question 6

Which value should be treated as a secret and kept out of client builds and source code repositories?

A. Public event title  
B. Leaderboard display name  
C. Token signing secret  
D. Button label text

**Answer: C**

**Explanation:**  
A token signing secret is a sensitive value. It should not be embedded in clients or committed to source code repositories.

### Question 7

Which example best shows server authority?

A. The client chooses the local sound volume.  
B. The client plays a button animation immediately.  
C. The server verifies whether a reward can be granted before changing inventory.  
D. The client changes the selected UI tab.

**Answer: C**

**Explanation:**  
Server authority means important outcomes such as rewards, currency changes, scores, and match results should be verified or decided on the server side.
